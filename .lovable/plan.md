
# Redesign Order/Sale detail screens

Apply the layout from the reference screenshots to **both** detail pages — mirrored between buyer and supplier (buyer shows Supplier name + "Message supplier"; supplier shows Buyer name + "Message buyer").

## Scope

- `src/pages/buyer/OrderDetail.tsx` (route `/buyer/orders/:id`)
- `src/pages/supplier/SaleDetail.tsx` (route `/supplier/sales/:id`)
- New shared CSS: `src/styles/mundus-deal-detail.css` (imported once in `index.css`)
- New shared components in `src/components/mundus/deal/`:
  - `DealHeaderCard.tsx` — header with icon, "#0000089", meta line (date · party · FCL · kg), status pill + total on the right
  - `PartyCard.tsx` — sidebar card (avatar w/ initials, name, country flag, "Message …" button)
  - `QuickTimeline.tsx` — Created / Pre-payment / Production / Shipment / Delivery with per-stage status
  - `NegotiationRoundsCard.tsx` — buyer-bid (blue) and counter (rose) bars per round, ACCEPTED badge
  - `NegotiationTotalsCard.tsx` — initial / first bid / discount / final + "Open full negotiation" link
  - `ShipmentRouteCard.tsx` — origin → destination with ETD/ETA + "Track on carrier"
  - `ShipmentJourney.tsx` — 7-stage horizontal stepper (Booked → Delivered)
  - `DocumentsPanel.tsx` — auto-generated docs row, progress bar, grouped categories (Trade / Shipping / Quality / Compliance) with REQUIRED tags + View/Download/Upload
- Extend `src/hooks/useBuyerOrders.ts` and `src/data/mockSales.ts` types (additive) with:
  - `negotiation?: { initialAskingUsd, firstBuyerBidUsd, rounds: [{ buyerBidUsd, counterUsd, accepted?, at }] }`
  - `shipmentDetails?: { containerNo, sealNo, booking, blNumber, vessel, voyage, portLoading, portDischarge, etd, eta, grossKg, netKg, temperature }[]`
  - `journeyStage?: 'booked' | 'loading' | 'departed' | 'in_transit' | 'arrived' | 'discharged' | 'delivered'`
  - `documents?: { autoGenerated: {...}, items: [{ id, category, name, required, uploadedAt?, sizeKb?, fileName? }] }`
  - Populate the existing 10 buyer orders and 25 supplier sales with believable mock data (rounds count varies by status, documents progress varies by status).
- i18n: add `buyer.orderDetail.v2.*` and `supplier.sales.detail.v2.*` blocks covering all new labels in en/pt/es.

- Tabs Negotiation / Shipment / Documents swap the left column content; right column stays the same on Overview but becomes ROUTE on Shipment.
- Mobile (<1100px): sidebar stacks below; tabs become horizontally scrollable; journey stepper scrolls horizontally.

## Mirroring rule

- `OrderDetail` (buyer) renders `<PartyCard role="supplier" name={order.supplierName} />` and "Message supplier".
- `SaleDetail` (supplier) renders `<PartyCard role="buyer" name={sale.buyer} />` and "Message buyer".
- All other sections are identical — driven by the same components fed from each page's hook.

## Acceptance criteria

1. `/buyer/orders/:id` and `/supplier/sales/:id` use the new layout matching the screenshots.
2. Four tabs render: Overview, Negotiation, Shipment, Documents (with `n/total` badge).
3. Header card shows order number, meta line, status pill, total value.
4. Right sidebar: party card + quick timeline on Overview/Negotiation/Documents; route card on Shipment.
5. Negotiation tab shows rounds with blue (bid) + rose (counter) bars and an ACCEPTED badge on the closing round.
6. Shipment tab shows per-container info + 7-stage journey stepper highlighted to current `journeyStage`.
7. Documents tab shows auto-generated SO/PO row, progress bar, and 4 grouped categories with REQUIRED tags and View/Download/Upload buttons (stubs → toast).
8. Buyer view shows supplier; supplier view shows buyer; both have a "Message …" button (stub → toast).
9. PT/ES translate cleanly, no missing keys.
10. No other buyer/supplier page is touched.

## Out of scope

- Real document uploads, real chat wiring, real negotiation API — all CTAs are toast stubs.
- Supplier `SupplierNegotiationDetail` and buyer `BuyerNegotiationDetail` stay untouched; the new Negotiation tab is a summary view with an "Open full negotiation" link to those pages.
